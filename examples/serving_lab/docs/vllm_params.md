# vLLM 参数速查

### `--model`

**作用：** 指向模型目录或 Hugging Face 模型名。

**影响：** 决定加载的权重、config、tokenizer 和 chat template。

**常见建议：** 生产环境优先使用本地固定路径，避免启动时临时下载。

### `--served-model-name`

**作用：** OpenAI-Compatible API 中客户端传入的 `model` 名称。

**影响：** 客户端请求必须和该名称匹配。

**常见建议：** 使用短名称，例如 `qwen`、`deepseek`、`glm`。

### `--host`

**作用：** 服务监听地址。

**影响：** `127.0.0.1` 仅本机访问，`0.0.0.0` 可被外部访问。

**常见建议：** 本地调试用 `127.0.0.1`，容器或远程服务用 `0.0.0.0`。

### `--port`

**作用：** 服务监听端口。

**影响：** 端口占用会导致启动失败。

**常见建议：** 默认 `8000`，多实例部署时显式区分端口。

### `--dtype`

**作用：** 控制权重和计算精度。

**影响：** 影响显存、吞吐和数值兼容性。

**常见建议：** NVIDIA Ampere 及更新 GPU 优先试 `bfloat16`，否则试
`float16` 或 `auto`。

### `--max-model-len`

**作用：** 设置服务允许的最大上下文长度。

**影响：** 直接影响 KV Cache 预算和最大可承载并发。

**常见建议：** 不要盲目设到 128K；先按真实业务长度设置。

### `--gpu-memory-utilization`

**作用：** 控制 vLLM 可使用的 GPU 显存比例。

**影响：** 太低会浪费显存，太高容易 OOM 或碎片化失败。

**常见建议：** 从 `0.85` 到 `0.9` 开始，稳定后再微调。

### `--kv-cache-memory-bytes`

**作用：** 显式指定 KV Cache 显存预算。

**影响：** 覆盖自动估算，更适合固定资源规划。

**常见建议：** 只有明确显存预算时使用，否则先用
`--gpu-memory-utilization`。

### `--max-num-seqs`

**作用：** 控制同时调度的最大序列数。

**影响：** 提升并发上限，也会增加 KV Cache 压力。

**常见建议：** 从 16、32、64 逐步压测。

### `--max-num-batched-tokens`

**作用：** 控制一次 batch 中可调度的 token 总数。

**影响：** 影响 prefill 吞吐、延迟和显存占用。

**常见建议：** 长 prompt 场景需要更谨慎，结合 P95/P99 调整。

### `--tensor-parallel-size`

**作用：** 使用多张 GPU 做 tensor parallel。

**影响：** 可承载更大模型，但增加通信开销。

**常见建议：** 大模型按 GPU 数和显存选择，单卡能跑时不必强行 TP。

### `--pipeline-parallel-size`

**作用：** 使用 pipeline parallel 切分模型层。

**影响：** 支持更大模型，但会引入流水线调度开销。

**常见建议：** 只有 tensor parallel 不足时再考虑。

### `--trust-remote-code`

**作用：** 允许加载模型仓库自定义 Python 代码。

**影响：** 某些 Qwen、GLM、DeepSeek 或特殊架构模型需要它。

**常见建议：** 只对可信模型开启，并固定模型版本。

### `--enable-prefix-caching`

**作用：** 开启前缀缓存复用。

**影响：** 多请求共享相同前缀时可降低 prefill 成本。

**常见建议：** RAG、固定 system prompt 和多轮对话场景值得测试。
